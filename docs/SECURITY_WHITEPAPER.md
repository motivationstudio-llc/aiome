# Aiome: セキュリティ仕様書 (Security Whitepaper)

## 0. はじめに
Aiome は、自律型 AI が直面する特有のセキュリティ脅威から、お客様の資産（APIクォータ、機密データ、稼働ログ）を守り抜くために設計されています。
従来の「境界型防御」ではなく、内部であっても信頼しない **「ゼロトラスト」** および **「多層防御（Defense in Depth）」** をアーキテクチャの根幹に据え、単一の脆弱性がシステム全体の崩壊を招くことのない強固な保護を実現しています。

---

## 1. The Abyss Vault (APIキー隔離保護)
APIキー（Gemini, OpenAI等）は、システムの生命線であり、最も狙われやすい資産です。Aiome では、このキーを「メインプロセスから物理的に隔離」します。

*   **プロキシ分離アーキテクチャ**: メインの AI エンジンプロセスは API キーを一切保持しません。通信はすべて隔離された `key-proxy` プロセスを経由します。
*   **メモリ空間のロックと消去**: `mlockall` によるスワップ禁止、および `zeroize` による使用済みメモリの強制上書きにより、ライブメモリ解析やディスクダンプ攻撃を無力化します。
*   **SSRF & リダイレクト防止**: 通信先はソースコードレベルでハードコードされており（ホワイトリスト制）、AI の誤操作や外部攻撃による不正なサーバーへのキー送信（SSRF）は構造的に発生しません。

## 2. ゼロトラスト実行環境 (WASM / Docker Shadow Worker)
Aiome が実行するスキル（動的コード）は、用途に応じて2種類の強力なサンドボックス内で動作します。

* **WASM Skills (軽量・高速)**
  * **完全なネットワーク隔離**: デフォルトではスキルから外界への通信は遮断されています。
  * **厳格なホスト許可制**: 実行に必要な通信先は、スキルごとに明示的にドメイン許可が必要です。ワイルドカード（`*`）による広域的なネットワークアクセスは Pro 版のランタイムで物理的に禁止されています。
  * **リソース制限（サーキットブレーカー）**: 無限ループや過度なメモリ消費に対しては、実行時タイムアウトとメモリキャップが作動し、システム全体の可用性を維持します。
* **Docker Shadow Worker (複雑タスクの委譲)**
  * **使い捨てコンテナ**: NPM パッケージや Python ライブラリなど、WASM 化できない重い依存関係が必要なタスクは、Docker コンテナとして一時的に切り出され、実行後は破棄されます。
  * **権限隔離**: コンテナ内の AI エージェントには `~/.ssh` や `aiome.db` などのホスト側の機密領域は一切マウントされません。
  * **MCP Client Lifecycle Defense**: 外部の MCP サーバー（Node.js, Python）を連携させる際、Aiome が終了すると「プロセスグループ (PGID)」単位で一斉キルシグナルが送信され、ゾンビプロセスやリソースの占有を防ぎます。

## 3. 不変の監査ログ (Immutable Audit Trails)
ログは「誰が何をしたか」を証明する唯一の手がかりです。Aiome では、このログ自体の改ざんを不可能にします。

*   **ハッシュチェーン技術**: SQLite データベース内のすべての実行ログは、一つ前のログのハッシュ値を取り込んだ暗号学的チェーンによって接続されています。
*   **改ざん検知**: ログの一行を消去したり、内容を書き換えたりするとチェーンが破綻するため、不正な操作を瞬時に検知可能です。
*   **自律進化の透明性**: AI が学んだ教訓（Karma）や、それに基づくシステム調整の履歴もすべてこの不変ログに記録され、AI の行動原理を完全に遡及・監査できます。

## 4. 堅牢な本番環境 (Secure Deployment)
運用環境そのものを攻撃ベクトルから守るため、コンテナレイヤーでのセキュリティを標準化しています。

*   **非特権（Rootless）実行**: Aiome は root 権限を一切必要としません。最小限の権限を持つ `aiome` ユーザーで動作し、コンテナ・エスケープ攻撃のリスクを最小化します。
*   **読み取り専用ファイルシステムへの対応**: 書き込み可能領域を最小限（ログとDBのみ）に限定し、ルートファイルシステムを読み取り専用でマウント可能な設計にしています。これにより、マルウェアの永続化を物理的に阻止します。
*   **ホストデーモンへの非直接アクセス**: AI がタスク委譲のために Docker を操作する際は、直接 UNIX ソケットを叩くのではなく、Native Tool 経由で制限された入力のみを受け付ける設計になっています。

---

## 結論
Aiome のセキュリティは、「隠すこと」ではなく「破られない構造を作ること」に重点を置いています。たとえ内部ソースコードが公開されたとしても、数学的・物理的、そして OS アーキテクチャ上の制約によって、お客様の API キーやデータの完全性は守られ続けます。
