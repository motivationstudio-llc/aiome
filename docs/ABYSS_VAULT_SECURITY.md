# The Abyss Vault (深淵の保管庫)
Aiome machine-level protection (The Abyss Vault) Security Architecture
本ドキュメントでは、エンタープライズ顧客およびセキュリティ担当者様向けに、Aiome がどのようにして「計算機科学上の絶対安全（APIキー漏洩の物理的・構造的不可能性）」を実現しているか、そのメカニズムを公開します。

---

### 1. 隔離された極限保護空間 (OSレベルでの隔離)
**「AI 自身を含め、誰もメインキーの存在を知らない」**

Aiome の中核となる自律型 AI プロセスは、**API キーを環境変数や設定ファイルとして一切保持していません。** 
すべてのプロンプト処理やエンベディングの要求は、隔離された別権限のプロセスとして動作する「Key Proxy（代理サーバー）」に対して投げられます。
たとえ万が一、AI のコアロジックが未知の脆弱性によって完全に乗っ取られたとしても、攻撃者が見つけられるのは「ローカルの Proxy に対して送信する権限」のみであり、クラウドインフラ（GCPやAWS）に直接アクセス可能な実体としての API キー（Secret）を奪取することは構造的に不可能です。

### 2. メモリ空間の物理ロック (スワップアウトの禁止)
**「ディスクに痕跡を残さず、メモリダンプ攻撃を無力化する」**

一般的なサーバー運用では、メモリ不足時に RAM の内容が HDD や SSD などのストレージ（スワップ領域）に書き出されます。攻撃者はこの仕様を悪用し、ストレージの残骸からメモリダンプを解析して API キーを窃取します。
The Abyss Vault では、起動時に OS のシステムコール (`mlockall`) を呼び出し、プロキシ自身のメモリ空間を物理 RAM に強制ロックします。これにより、**いかなる状況でも API キーが含まれるメモリ内容がストレージに漏出することをOSレベルで禁止**しています。権限不足などでこのロックが失敗した場合、システムは安全のために即座に起動を停止 (`panic!`) します。

### 3. 環境変数の即時自己消去 (Self-Wiping ENV)
**「起動から1ミリ秒後に、自身の記憶を消去する」**

コンテナ環境などで一時的に渡された環境変数（`GEMINI_API_KEY` 等）は、/proc ファイルシステムなどを経由して子プロセスから盗まれる危険性があります。
The Abyss Vault は、起動直後の 1 ミリ秒以内に環境変数から API キーを読み取ると、**即座に環境変数空間からその文字列を物理的に消去（ゼロ化）**します。これにより、外部からの環境変数覗き見攻撃や、依存パッケージ（ライブラリ）に潜むバックドアからの窃取を無効化します。

### 4. 実行時メモリの暗号学的消去 (Zeroize)
**「使用されたキーは、文字列として決して RAM 上に留まらない」**

HTTP リクエストを構築するために一時的にメモリ上に API キーが展開された直後、The Abyss Vault は `zeroize`（暗号的メモリ消去）技術を用いて、**使用済みバッファを強制的にゼロ（0x00）で上書き**します。一般的なガベージコレクション（GC）によるメモリ解放を待つのではなく、能動的に RAM 上の存在を抹消することで、ライブメモリ解析によるわずかな隙すら与えません。

### 5. ハードコードされた通信経路 (SSRF＆リダイレクトの封鎖)
**「AI がどこへ向かおうとも、辿り着くのは正規ルートのみ」**

AI（LLM）の出力や、外部からの攻撃によって「不正な宛先（攻撃者が用意したサーバー）」にリクエストを誘導させ、API キーを含んだヘッダーを送信させてしまう SSRF (Server-Side Request Forgery) 攻撃。
The Abyss Vault では、プロキシ内で行き先の URL が**ソースコード上にハードコード（Enum固定）**されています。AI がどんな URL を要求しようとも、通信は必ず Google や OpenAI などの公式プロバイダーの特定エンドポイントにのみルーティングされます。
さらに、HTTP 3xx 系の「リダイレクト追従」を機能レベルで禁止（No-Redirect Policy）しているため、プロバイダーを偽装した中間者攻撃（MITM）によるキーの横取りも不可能です。

### 6. 認証とレート制限によるゼロトラスト (Identity Verification)
**「味方であっても信頼しない」**

The Abyss Vault にアクセスできるのは、あらかじめ登録された「Aiome の正規コンポーネント（Daemon や Watchtower）」のみです。 Caller ID による検証を行い、未知のプロセスからのアクセスは即座に `403 Forbidden` として遮断されます。
また、API の過剰利用（DDoS や暴走による課金爆発）を防ぐため、全体およびコンポーネントごとの厳密なリクエスト上限（Quota）をプロキシ内でハードリミットとして管理しています。

## 結論

**Aiome は、お客様の大切な LLM クォータ（資産）をお預かりする上で「利便性のための妥協」を一切行いません。**

「The Abyss Vault」のアーキテクチャは、コードの難読化（隠蔽）で時間を稼ぐようなセキュリティ・バイ・オブスキュリティではなく、**「攻撃者にこの設計図の全貌とソースコードを公開した上で、なお数学・物理的、および OS のアーキテクチャ上突破が不可能である」**ことを前提として設計されています。

Aiome を導入することで、企業は「自社開発の AI アプリケーションから万が一 API キーが漏れるかもしれない」という最大のコンプライアンスリスク・事業リスクから完全に解放されます。

## 付属資料: アーキテクチャに関する FAQ

**Q: すべての外部 API (Web検索機能やDiscord通知ボットなど) が The Abyss Vault を経由していないのはなぜですか？保護範囲の漏れではありませんか？**

**A: いいえ、これは意図的な「可用性とセキュリティのバランス設計（マイクロサービスによる権限と障害ドメインの分離）」です。**

当社では、ゼロトラストにおける**「資産価値のレイヤリング（階層化）」**を採用しています。

1.  **致命的リスクの極限保護 (LLM API キー等)**
    最も高額で悪用価値が高く、漏洩時に青天井の従量課金被害（Compute Theft）が発生する恐れのあるメイン・プロバイダーキーは、最高警戒レベルの「The Abyss Vault」内に物理的に隔離し、外界との直接通信を遮断しています。
2.  **単一障害点（SPOF）の回避 (監視・通知系 API 等)**
    Discord ボットである `Watchtower` 等は、システムの監視とフェイルセーフな復旧（緊急停止コマンドなど）を担う独立したコンポーネントです。もし全ての通信を単一の Vault（プロキシ）に集約してしまうと、Vault 自体に障害が発生した際に**「管理者にアラートを送る手段（Discordボット）まで一緒に機能不全に陥る」という致命的なアーキテクチャ欠陥（SPOF）**を引き起こします。
3.  **ブレスト・ラディウス（被害範囲）の局所化**
    Web 検索 API や通知系 API は、仮に漏洩しても定額プランの上限に達する程度の軽微な被害（Blast Radius の矮小性）に留まります。これらはリスクレベルに応じて、AI コアとは別のライフサイクル・別の権限セットで動作するサービスとして設計されています。

エンタープライズ運用においては、無闇に単一の要塞に全てを詰め込むのではなく、「何を守り、システムがダウンした際にどうやって復旧経路を確保するか」という**レジリエンス（回復力）**への配慮が不可欠です。それこそが、強固な実運用に耐えうる当アーキテクチャの真の価値です。
